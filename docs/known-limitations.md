# SURI Known Limitations

**문서 목적**: PoC의 알려진 한계를 정직하게 기록. 면접 방어 탄약이자 Future Work 출발점.

**철학**: "할 수 있는 것과 할 수 없는 것을 구분하는 능력"이 "모든 걸 할 수 있다고 주장하는 능력"보다 중요. 면접에서 한계 질문 받으면 이 문서를 근거로 응답.

**최종 갱신**: 2026-04-23 (D-6) — ADR-004 Domain Glossary 도입 후 C1 3회 반복 측정 반영

---

## 1. 합성 데이터 한계

### 1.1 절대값 업계 대비 괴리

본 PoC는 100% 합성 데이터 (ADR-001 결정). 규모·분포는 설계대로 재현되나 **절대 수치는 업계 실측과 차이**.

| 지표 | 본 PoC 실측 | 업계 평균 (금감원 2024) |
|---|---|---|
| 13회차 유지율 | ~58% (전 채널 평균) | 88.2% (생보) |
| 25회차 유지율 | ~48% | 68.9% |
| 손해율 proxy | — | — (생보는 3이원 분해가 표준) |

**원인**: 합성 데이터 생성 시 의도 이상 패턴(GA 낙차, 3월 절판 스파이크)을 강조하면서 전반적 유지율이 낮게 설정됨. 업계 벤치마크 직접 복제가 목적이 아닌 **상대 패턴 재현**이 목적.

**대응**:
- CRITIC_SYSTEM에 "Data caveat" 문구 추가 — Agent가 절대값 언급 시 "합성 데이터라 업계와 괴리 가능"을 자동 명시
- 업계 참조 벤치마크는 Glossary `유지율.industry_benchmark` 필드로 분리 관리

**면접 방어**:
> *"합성 데이터의 강점은 **통제된 이상 패턴 재현**입니다. 실데이터처럼 완벽한 절대값 재현은 포기하고, GA 낙차 1위·3월 스파이크 같은 분석적 도전 과제를 정확히 주입했습니다. 이 덕분에 Agent가 '낙차 있는 채널을 식별'하는 분석 능력을 결정적으로 검증할 수 있습니다."*

### 1.2 의도 주입 이상 패턴 3가지

다음 패턴은 합성 데이터에 의도적으로 주입 (검증 가능):

| 패턴 | 내용 | 재현 여부 |
|---|---|---|
| GA 채널 유지율 낙차 | 13·25회차 lapse probability 19% overlay | ✅ 재현 (3/3 측정) |
| 3월 절판 스파이크 | 평월의 1.8배 신계약 | ✅ 재현 (C2 시나리오) |
| 특정 product_id 25회차 이상치 | 교보생명 2023 사례 모델 | ✅ 재현 (C8 시나리오) |

상세: `docs/seed-plan.md`

## 2. Agent 실행 비결정성

### 2.1 C1 3회 반복 측정 결과 (2026-04-23 D-6)

Glossary 도입 후 C1 (채널별 유지율 낙차) 3회 반복 측정:

| Run | 낙차 1위 | GA 낙차 | SQL CTE 구조 |
|---|---|---|---|
| 1 | GA | 1.07%p | `cohort_13/25, survived_13/25` |
| 2 | GA | 7.92%p | `cohort_13/25, survived_13/25, valid_policies` |
| 3 | GA | 17.88%p | `cohort, paid_13/25, channel_stats` |

### 2.2 결정성 수준 분석

**결정적 (3/3)**:
- 상대 순서: GA가 낙차 1위
- 의미론적 정의: 코호트 기반 분모 (glossary 주입)
- 구조적 함정 회피: `payment_month_seq >= N`을 분모로 잡는 오류 0건

**비결정적 (편차)**:
- 낙차 절대값 (1.07 → 17.88%p 편차)
- SQL CTE 구성 (매번 다른 이름·구조)
- WHERE 조건 세부 (일부 Run은 `valid_policies` 등 추가 필터)

### 2.3 해석

**Glossary의 책임**: 용어 **정의** 결정성.
**Glossary의 비책임**: SQL 구조 결정성.

SQL 템플릿을 프롬프트에 박으면 이 편차를 제거 가능하나:
- Agent의 reasoning/planning 자율성 훼손
- 골든셋 C1만 해결하는 overfitting 위험
- 다른 질문 변형 대응력 약화

**의도적 선택**: 절대값 편차는 Agent 자율성 영역으로 남김. 상대 순서만 보장.

**면접 방어**:
> *"Glossary는 '정의'를 결정적으로 주입하고, SQL 구조는 Agent가 자율적으로 설계합니다. 3회 측정에서 CTE 구성은 매번 다르지만 모두 코호트 기반 정의를 따르며 GA가 낙차 1위라는 결론은 동일합니다. 이는 Agent가 정의를 결정적으로 받되 reasoning은 자율적으로 수행한다는 증거입니다. SQL 템플릿을 박으면 이 편차를 0으로 만들 수 있지만, 그건 Agent가 아니라 스크립트입니다."*

## 3. IFRS17 지표 산출 불가

### 3.1 산출 불가 지표 목록

| 지표 | 불가 사유 | Glossary 대응 |
|---|---|---|
| CSM | BEL + RA + 할인율 체계 부재 | `suri_산출_가능_여부: "불가"` + `권장_대응_graceful_degradation` |
| BEL | 계리 가정·할인율 부재 | 동일 |
| RA | 신뢰수준법 기반 분위 차감 체계 부재 | 동일 |
| 해약환급금 | 책임준비금·사업비 구조 부재 | 동일 |
| 위험손해율 | 위험보험료 분리(Unbundling) 데이터 부재 | `지급률_proxy`로 대체 |

### 3.2 대응 패턴 (graceful degradation)

사용자가 이 지표 질문 시 Agent 동작:
1. Glossary 조회 → `산출_가능_여부: 불가` 확인
2. 불가 사유 명시 (예: "BEL 데이터 부재")
3. **대안 제시**:
   - CSM → 보장성 상품 APE 규모 (상관 proxy)
   - 위험손해율 → 지급률 proxy (명시적 명칭 구분)
4. 정확 수치가 필요하면 계리팀 산출 권고

**면접 방어 (C7 시나리오)**:
> *"CSM이나 손해율 같은 IFRS17 결산 지표는 본 PoC 스키마로 산출 불가합니다. BEL·RA·할인율 체계가 없기 때문입니다. Agent는 이를 Glossary로 결정적으로 인지하고 proxy 지표를 대안으로 제시하거나 '계리팀 산출 권고'로 응답합니다. 이게 hallucination 방어의 핵심 장치입니다."*

## 4. 데이터 스키마 한계

### 4.1 세부 컬럼 부재

| 누락 항목 | 영향 |
|---|---|
| 해약환급금 금액 | 조기해지 유인 분석 불가 |
| 책임준비금 | BEL·CSM 산출 불가 |
| 사업비 구조 | 정식 손해율 산출 불가 |
| 위험보험료 분리 | 생보 위험손해율 산출 불가 |
| 측정모형별 계리 가정 | IFRS17 산출 불가 |

### 4.2 해지 상태값 혼재

| 실무 구분 | 본 PoC `status` | 혼재 |
|---|---|---|
| 자발적 해지 | `cancelled` | 정확 |
| 보험료 미납 실효 | `lapsed` | 실효/실효해지/부활 미구분 |
| 고지의무 위반 무효 | `void` | 정확 |

**영향**: 조기해지율 분석에서 "미납 실효"와 "실효해지"를 구분 불가. 실무 상세 분석 제한.

**대응**: Glossary `해지_및_실효.suri_스키마_매핑`에 명시.

## 5. 두 유지율 정의의 공존

### 5.1 seed/checks.py vs Agent SQL 불일치

| 컴포넌트 | 유지율 계산 방식 |
|---|---|
| `seed/checks.py` | `COUNT(status='active') / COUNT(*)` (현재 상태 기반) |
| Agent 생성 SQL (glossary 기반) | `premium_payments_v.payment_month_seq` 기반 코호트 계산 |

두 방식은 **정의가 다름**. 같은 데이터에 대해 다른 수치를 낸다. checks.py는 seed 검증용 빠른 sanity check, Agent는 업계 표준 정의 적용.

**면접 방어**:
> *"두 방식이 공존하는 이유는 역할이 다르기 때문입니다. checks.py는 합성 데이터 생성 직후 데이터 무결성을 빠르게 검증하는 도구이고, Agent SQL은 업계 표준(금감원 2023 APE 기준)을 따라 쿼리를 생성합니다. 프로덕션 이관 시 checks.py는 제거하고 Agent 기준만 유지합니다."*

## 6. Agent/UI 기능 한계

### 6.1 멀티턴 대화 미지원

- Planner에 이전 turn 맥락 전달 구조 없음
- Scene 3 (드릴다운 시나리오)는 UI 프리셋으로만 제공, 실제 멀티턴 reasoning은 미구현
- D-5 작업 계획에 포함

### 6.2 실시간 스트리밍 UI 미구현

- 현재 UI는 파이프라인 완료 후 전체 결과를 한 번에 렌더
- Planner 완료 → Executor 실행 중 → Critic 해석 단계별 렌더는 미구현
- A2 옵션으로 검토했으나 D-6 시점 우선순위에서 후순위로 이관

### 6.3 Self-correction 최대 2회 제한

- Executor가 GuardViolation 또는 DBExecutionError 시 최대 2회 재시도
- 2회 초과 시 에러 결과를 그대로 반환 (추가 재시도 없음)
- `MAX_GUARD_RETRIES = 2` 상수로 관리
