# SURI 면접 답변 템플릿

**원칙**
- 솔직함 > 연기 (AI 사용 인정, 대신 "감독·검증" 포지셔닝)
- 30초 answer + 15초 증빙 + 30초 확장 (총 75초 이내)
- 모든 답변은 **구체 예시 1개 이상** 포함

---

## A. 오프닝

### A1. "프로젝트 간단히 소개해주세요"

> **30초**: "SURI는 한화생명 Data Agent Developer 포지션을 겨냥한 9일 PoC입니다. 보험 상품 기획·분석 담당자가 자연어로 질문하면, 3-Agent(Planner·Executor·Critic)가 협업해서 PostgreSQL에 SQL 쿼리를 생성·실행하는 시스템입니다."
>
> **15초 차별화**: "핵심 차별점은 MCP 기반 PII 거버넌스 3층 방어입니다. Application Layer의 SQL Guard, DB Layer의 읽기전용 ROLE, Schema Layer의 VIEW까지 다층으로 쌓았습니다."
>
> **30초 시연 유도**: "15분쯤 보여드릴 수 있는데, 가장 인상적인 건 Agent가 의도적으로 PII 테이블을 쿼리해도 3층에서 각각 차단되는 걸 로그로 보실 수 있다는 점입니다."

---

## B. 역할·AI 활용 질문 (이 섹션이 가장 중요)

### B1. "이 프로젝트 어떻게 개발하셨어요?" (가장 흔함)

> **AI 인정 + 포지셔닝**: "Claude Code와 Claude 웹 채팅을 적극 활용했습니다. 역할을 명확히 나눠서, AI는 구현 초안을 담당하고, 저는 설계 판단·검증·의사결정을 담당했습니다."
>
> **구체 예시 1**: "예를 들어 MCP 라이브러리 선택 시, 초기에는 `fastmcp`(커뮤니티 래퍼)가 추천됐는데, 금융권 이관 관점에서 의심이 들어 4개 AI에 교차 검증을 요청했습니다. 그 과정에서 FastMCP 1.0이 공식 SDK 내부에 편입되어 있다는 사실을 발견했고, 최종적으로 공식 SDK(옵션 A2)를 선택했습니다. ADR-002에 근거 정리했습니다."
>
> **구체 예시 2**: "데이터 모델링에서 `payment_month_seq`를 역정규화 저장하자는 초안이 있었는데, 역산 리스크를 지적해서 VIEW 기반 동적 계산으로 바꿨습니다."
>
> **마무리**: "즉 AI가 **초안을 쓰고**, 저는 **그 초안의 전제와 리스크를 심사**했습니다. 9일 타임박스에서 이게 시니어 역할에 더 가깝다고 판단했습니다."

---

### B2. "그럼 진짜 본인이 짠 건 뭐예요?" (압박성)

> **솔직 인정**: "코드 라인 수로는 AI가 70% 이상입니다. 근데 그게 부끄럽지 않습니다."
>
> **4가지 주도 영역**:
> 1. **기획·포지셔닝**: JD 분석, 차별화 축 선정 (MCP + PII 거버넌스)
> 2. **의사결정**: 매 초안마다 검증·반박·교정. 4번의 교차검증 리서치로 설계 확정
> 3. **도메인 지식**: KIRI 유지율 수치, IFRS 17, 한국 PIPA 구분, 한화생명 채널 맥락
> 4. **검증 루프**: 각 실행 결과 분석해서 다음 방향 결정
>
> **마무리**: "극단적으로 말하면 CTO 역할을 제가 하고 AI가 시니어 엔지니어 역할을 했습니다. 결정 권한으로 보면 100% 제 프로젝트입니다."

---

### B3. "요즘 AI 도구 없이도 코딩 가능하세요?" (역량 의심)

> **명확 인정**: "AI 도구 없이도 코딩 가능합니다. 근데 2026년 시점에 AI 없이 혼자 9일에 이만큼 만드는 건 비효율이라고 판단했습니다."
>
> **근거**: "기초적인 Python, SQL, Docker, PostgreSQL 운용은 제가 직접 이해하고 있습니다. 이 프로젝트 안에서 한 예를 들면, INTEGER → BIGINT 전환 시 VIEW 의존성 때문에 ALTER가 막혔는데, DROP VIEW → ALTER → CREATE VIEW 패턴을 제가 직접 구성했습니다. PostgreSQL dependency tracking 이해가 있어야 가능한 판단이었습니다."
>
> **역질문 가능**: "혹시 제가 직접 짜야 한다고 보시는 부분이 있으면 화이트보드에서 라이브로 풀어드릴 수 있습니다."

---

### B4. "AI가 만든 코드 버그 나면 어떻게 해요?"

> **접근법**: "저는 코드가 동작하는 것과 코드가 **왜** 그렇게 동작하는지를 구분합니다. AI 초안은 동작은 하지만 근거는 설명 필요합니다. 그래서 매 파일마다 제가 구조 설명을 받고 이해한 뒤에 승인합니다."
>
> **구체 예시**: "`guards.py` 작성 시 sqlglot AST 기반 접근이 왜 정규식보다 강한지 4가지 구체 케이스(별칭, CTE, 스키마 접두사, 문자열 오탐)를 확인한 후 통과시켰습니다. 면접에서 이 부분 더 깊이 물으시면 대답 가능합니다."
>
> **디버깅 사례**: "실제로 Feb 29 버그, INTEGER overflow, 상대 임포트 이슈 등 3가지 이상 제가 직접 진단해서 수정 방향을 결정했습니다."

## C. 기술 심화 질문

### C1. "MCP가 뭐고 왜 썼어요?"

> **정의**: "MCP는 Model Context Protocol로 Anthropic이 2024년 공개한 Agent-Tool 표준 프로토콜입니다. Agent가 외부 도구(DB, API, 파일시스템 등)를 호출할 때의 규약입니다."
>
> **왜 선택**: "이 JD에 'MCP 연동 능력' 키워드가 명시됐고, 이게 SURI의 차별화 축이기도 합니다. Function calling보다 구조적 이점이 있습니다. Agent 구현과 도구 인터페이스가 분리되어, 나중에 Claude 외 다른 LLM으로 교체해도 MCP tool은 그대로 재사용 가능합니다."
>
> **구체 구현**: "`execute_readonly_sql` tool 하나를 중심으로, FastMCP 데코레이터로 JSON 스키마 자동 생성, 타입 힌트 기반입니다. MCP Inspector로 프로토콜 레벨 검증도 마쳤습니다."

### C2. "3층 방어 구조 설명해주세요"

> **개요**: "Defense in Depth 원칙 적용. 한 층이 뚫려도 다음 층이 백업합니다."
>
> **각 층**:
> 1. **Layer 1 - Application**: SQL Guard. sqlglot AST로 SELECT 검증, `customers` 테이블 차단, multi-statement 탐지
> 2. **Layer 2 - DB ROLE**: `suri_readonly` 계정. GRANT SELECT만, UPDATE/DELETE/DROP 권한 없음
> 3. **Layer 3 - Schema**: `customers_safe` VIEW만 공개. PII 원본 컬럼(`resident_number`, `phone`, `email`) 제외
>
> **시연**: "데모에서 의도적으로 Guard 우회한 SQL을 직접 DB로 보내면 Layer 2가 `permission denied`로 차단하는 것을 보여드립니다. 각 층이 독립 작동함을 증명합니다."

### C3. "SQL 파싱에 왜 AST? 정규식으로 충분 아닌가요?"

> **정규식의 한계 4가지**:
> 1. 별칭 우회: `FROM customers c WHERE c.gender...`
> 2. 스키마 접두사: `FROM public.customers`
> 3. CTE 중첩: `WITH c AS (SELECT * FROM customers) SELECT * FROM c`
> 4. 문자열 오탐: `SELECT 'I love customers' FROM products`
>
> **AST 해결**: "sqlglot의 `find_all(exp.Table)`이 트리 전체 재귀 순회. 별칭·CTE·서브쿼리 내부까지 `exp.Table` 노드 전부 찾음. `.name` 속성이 테이블 원본 이름이라 별칭 우회 자동 방어."
>
> **맥락 강조**: "Agent가 만드는 SQL은 예측 불가능합니다. 한 번 뚫리면 거버넌스 서사 전체가 무너집니다. 표면 패턴 매칭 아닌 의미 단위 검증이 필요합니다."

### C4. "합성 데이터 썼는데 현실성은?"

> **의도 설명**: "SURI의 핵심은 데이터 현실성이 아닌 거버넌스 시연입니다. 따라서 데이터는 **통제 가능한 합성**이 최적이었습니다."
>
> **근거**:
> 1. 실제 한국 PIPA상 합성데이터는 개인정보 아님 → 법적 리스크 없음
> 2. KIRI 자료 기반 이상 패턴 3개(GA 유지율, 3월 스파이크, 특정 상품 이상치) 의도적 주입
> 3. 시드 고정으로 재현성 100%
>
> **수치 증명**: "예를 들어 GA 채널 25회차 유지율이 81.3%인데, 이건 KIRI 2022 보고서의 업계 평균 낙차(19%p)를 의도적으로 재현한 것입니다."

### C5. "이 프로젝트 프로덕션 이관 가능해요?"

> **정직**: "9일 PoC라 그대로는 아닙니다. 3가지가 필요합니다."
>
> **Gap 분석**:
> 1. **인증/권한**: 지금 `.env` 기반인데 프로덕션은 secret manager(AWS Secrets Manager, Vault 등) 연동 필요
> 2. **스케일링**: 현재 PostgreSQL 단일 인스턴스. Read replica + Redis 캐시 필요
> 3. **관측성**: JSONL 로그만 있음. DataDog, ELK 등 실제 APM 연동 필요
>
> **재사용 가능한 부분**: "스키마 설계, PII 거버넌스 구조, MCP tool 인터페이스는 그대로 이관 가능합니다. 특히 의존성을 `mcp` 단일 패키지로 유지한 게 사내 OSS 심사에 유리합니다."

## D. 압박성 질문

### D1. "ChatGPT한테 프로젝트 짜달라고 하면 이거랑 비슷할 텐데요?"

> **차분 대응**: "네, 구조 자체는 비슷할 수 있습니다. 근데 중요한 건 두 가지입니다."
>
> **차별점 1 - 의사결정**: "예를 들어 MCP 라이브러리 선택은 ChatGPT가 추천한 `fastmcp`(독립 프로젝트)를 받아들이지 않고, 금융권 이관 관점을 반영해서 공식 SDK로 바꿨습니다. 이 과정이 ADR-002에 문서화되어 있습니다."
>
> **차별점 2 - 도메인 지식**: "데이터 분포는 ChatGPT가 준 걸 그대로 안 썼습니다. KIRI 수입보험료 기준과 한화생명 자회사 GA 맥락을 별도로 조사해서 '시장 평균 baseline + GA subset에 이상 패턴 주입'으로 설계했습니다. 이건 ChatGPT가 말해주지 않는 부분입니다."
>
> **종결**: "AI는 표준 솔루션을 주고, 저는 이 회사 이 포지션 이 맥락에 맞게 판단했습니다."

### D2. "9일이면 AI 없으면 못 하는 거 아닌가요?"

> **전환**: "맞습니다. 그게 요점입니다. 2026년 현재 시니어 엔지니어는 AI 없이 일하는 사람이 아니라 AI를 **잘 쓰는** 사람입니다."
>
> **비유**: "계산기 없던 시절 수학자와 계산기 쓰는 수학자 중 누가 더 많은 문제를 푸나요? 후자입니다. 중요한 건 어떤 문제를 풀 가치가 있는지, 계산 결과가 맞는지 판단하는 능력입니다."
>
> **SURI 맥락**: "저는 이 프로젝트에서 9일에 혼자 구현할 수 없는 범위를 목표로 삼았습니다. AI 없이 9일에 가능한 범위로 축소했다면, MCP나 3층 거버넌스 같은 차별화를 포기해야 했을 겁니다. 그건 포지션에 덜 적합한 결과였을 거고요."

### D3. "이 코드 설명해보세요 (아무 파일 지적)"

> **대응 원칙**: "당황 말고 파일 먼저 열어서 보고 말함. 준비 안 된 척하는 게 더 자연스러움."
>
> **답변 구조**:
> 1. 파일 목적 1문장
> 2. 핵심 함수 1~2개 설명 (파라미터 이름, 반환값)
> 3. **왜 이렇게 설계했는지** (설계 근거 카드 내용)
>
> **실패 대응**: 디테일 모르면 "이 라인은 지금 바로 설명 어려운데, 저희 docs/adr에 근거 문서가 있습니다. 라이브로 열어봐도 될까요?" → 오히려 투명함 어필

### D4. "시연 중 에러 나면?"

> **대응**:
> 1. 당황하지 않기. "이건 실제 개발 환경 이슈인데 잠깐 로그 볼게요"
> 2. stderr 로그 보여주면서 Layer 판단 — "이건 Layer X에서 발생했고..."
> 3. 해결 가능하면 고치기 / 아니면 우회 시연
>
> **Fallback**: 미리 **녹화 영상** 준비. "실시간 데모 중 네트워크 이슈가 있을 수 있어 녹화본도 준비했습니다"
