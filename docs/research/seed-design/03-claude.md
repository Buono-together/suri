## Q1. 파일 구조·실행 방식

**결론: 2개 파일 추가 필요, 나머지 OK.**

```
data/seed/
├── __init__.py
├── config.py           # 파라미터만 (dict/dataclass)
├── distributions.py    # ★ 추가: 분포 샘플링 헬퍼 (연령·상품군·채널)
├── anomalies.py        # ★ 추가: 이상 패턴 주입 (면접 하이라이트)
├── generators.py       # 엔티티별 순수 함수
└── load.py             # DB I/O
```

**`anomalies.py`를 분리해야 하는 이유**: 면접 중 화면 공유로 *"이상 패턴은 이 50줄 모듈에 집중되어 있고, 파라미터는 여기서 토글됩니다"*라고 보여줄 수 있어야 합니다. `generators.py`에 섞여 있으면 설명 시간이 늘어나고 전달력이 떨어집니다.

**실행 방식 보완점**:
- TRUNCATE 시 `TRUNCATE ... RESTART IDENTITY CASCADE` 명시
- `--dry-run` 플래그 (DB 연결 없이 샘플 1000행만 생성해서 분포 검증용)
- 실행 후 자동 검증: 채널별 유지율, 3월 절판 비율 등 **anomaly가 실제로 주입됐는지 SELECT로 assert**. 면접에서 "검증까지 스크립트에 포함"은 강한 신호.

## Q2. 생성 순서 & 성능 (P0 이슈 포함)

순서는 정확합니다. 문제는 **성능**입니다.

### ❌ 현재 예상되는 naive 경로
```python
cur.executemany("INSERT INTO premium_payments VALUES (...)", rows)
```
→ 200k 행에 5~10분. PoC에서 데이터 재생성 시 반복 실험 불가.

### ✅ psycopg 3의 COPY protocol 필수
```python
with cur.copy(
    "COPY premium_payments (policy_id, payment_date, amount, ...) FROM STDIN"
) as copy:
    for row in rows:
        copy.write_row(row)  # 200k rows in ~2-5 seconds
```
**10~50배 빠름**. 단일 트랜잭션으로 감싸면 더 빨라집니다.

### 추가 성능 권장
- **전체를 단일 트랜잭션**으로 (`conn.autocommit = False`)
- **FK/인덱스 deferred**: 200k 규모면 효과 미미하지만 면접에서 "대용량 bulk load 의식"을 보여주는 장치로 값어치 있음
- **`ANALYZE` 실행**: 로드 직후 실행해서 통계 업데이트 (Agent 쿼리 플랜이 합리적이 되도록)
- **`claims`도 5k라 작지만 COPY 일관성 위해 동일 방식** 권장

### 추가 단계 누락
생성 순서에 **"6.5. VIEW 권한 부여 + 검증 쿼리"** 단계가 빠져있습니다. `customers_safe` VIEW의 `suri_readonly` 권한이 실제로 작동하는지 로드 직후 자동 체크해야 3층 방어 시연이 깨지지 않습니다.

## Q3. PII 처리 (한국 특화 이슈)

설계안은 80% 완성. 누락 포인트 4가지:

### 3.1 주민번호: **`RRN-XXXXXX` 더미 포맷 권장** (체크디짓 수정 안 함)
체크디짓만 틀리게 하면 여전히 DLP 정규식 `\d{6}-[1-4]\d{6}`에 걸립니다. 완전 비-주민번호 포맷이 안전:
```python
rrn = f"RRN-{fake.random_number(digits=6):06d}"
```
면접에서 *"체크디짓 오류는 포맷은 맞추기 때문에 DLP에 여전히 노출됩니다. 저는 포맷 자체를 어긋내는 방식을 선택했습니다"* → 1단계 깊은 답변.

### 3.2 전화번호: **Faker 기본값 위험**
`Faker('ko_KR')`의 phone은 실제 한국 이통사 번호 대역을 포함할 수 있습니다 (010-3000-XXXX 등 실존). **의도적으로 invalid prefix 사용**:
```python
phone = f"010-99{fake.random_number(digits=2):02d}-{fake.random_number(digits=4):04d}"
# 010-99XX-XXXX: 이통사 미할당 대역
```

### 3.3 주소: **`시도 + 시군구`로 truncate**하되 실존 시군구 매핑
`fake.address()`의 상세주소는 버리되 `fake.city()` + `fake.administrative_unit()` 조합으로 재구성. 더 자연스럽고 presenter가 보기 깔끔.

### 3.4 추가 고려: **`customer_id` 자체의 PII성**
`customer_id`를 `c_00001` 같은 순차 값이 아닌 **UUID 또는 SHA256 해시**로. 순차 ID는 "가입 순서"를 노출시켜 준-PII로 취급될 수 있습니다. 한화생명 면접관은 이걸 봅니다.

### 3.5 **누락**: 생년월일(dob) 마스킹 전략
`customers`에 dob 컬럼 있다면 이것도 PII. `customers_safe`에서는 `age_band` (10세 단위)로만 노출해야 주민번호 앞 6자리 역산 차단 스토리가 완성됩니다.

## Q4. 관계형 정합성 (P0 이슈 포함)

현재 룰은 구조적 정합성에 집중되어 있습니다. **"Agent가 SQL로 검증 쿼리를 날렸을 때 이상하지 않아야 한다"** 축이 빠져있습니다.

### ★ 필수 추가 룰 (누락 시 데모 중 catch될 가능성 높음)

**4.1 상품-연령 정합성** (P0)
```python
product_age_rules = {
    "어린이보험": (0, 15),
    "연금": (40, 65),
    "종신": (30, 55),
    "건강":(25, 70),
    "저축": (30, 60),
    "정기": (30, 55),
}
```
이게 없으면 "60대 어린이보험 가입자" 같은 outlier가 생성됩니다.

**4.2 계약 금액 ↔ 상품군 정합성**
- 종신 월 보험료 20~50만원 범위
- 건강 3~15만원
- 연금 20~100만원 (일시납 포함)
- 정기 1~10만원

**4.3 납입주기 ↔ 상품유형 조합 제약**
- 변액 상품 → 일시납 없음 (월납/연납만)
- 정기 → 일시납 없음
- 저축성 → 일시납 비중 높음 (30%)

**4.4 상태 전이 시간 논리**
- `lapsed` 계약: `last_payment_date` + 2개월 유예기간 = `lapsed_date`
- `surrendered` vs `lapsed` 구분 (자발적 해약 vs 실효)
- 청약철회 (15일 이내): 전체의 1~2%

**4.5 에이전트-계약 분포 (Pareto)**
모든 agent가 균등하게 계약을 가지면 비현실적:
```python
# 상위 20% agent가 전체 계약의 60~70% 담당 (파레토)
agent_weights = np.random.pareto(1.2, n_agents)
```

**4.6 보험금 청구 로직**
- 계약 후 면책기간 (통상 90일) 내 claim 없음
- 종신·건강 상품에만 claim 집중 (저축성/연금은 claim 거의 없음)
- `claim_date`는 반드시 `issue_date + 90일` 이후

**4.7 합계 정합성**
`policies.total_premium_paid` = `SUM(premium_payments.amount)` 일치해야 Agent의 집계 검증 쿼리가 통과.

## Q6. 이상 패턴 주입 방식

**결론: 2단계(기본 생성 → overlay) 방식 강력 권장.**

### 각 방식의 장단점

| 방식 | 장점 | 단점 | 이 PoC 적합도 |
|---|---|---|---|
| 1. if-else | 구현 단순 | 로직 얽힘, 이상 패턴 찾기 어려움 | ❌ 면접 설명 난이도↑ |
| 2. 가중치 샘플링 | 분포 매칭 자연스러움 | 명시적 스파이크 표현 어려움 | △ 절판 스파이크 표현 약함 |
| 3. **2단계 overlay** | **분리·토글·검증 쉬움** | 구현 공수 약간 증가 | ◎ **최적** |

### 구체 구현 패턴

```python
# generators.py — 기본 생성 (이상 패턴 無)
def generate_policies(customers, products, agents, config):
    return [_make_policy(c, p, a) for ...]  # 균질 분포

# anomalies.py — 2단계 overlay
def inject_ga_retention_drop(policies, rate=0.19, seed=...):
    """GA 채널 계약 중 25회차 시점에 19% 확률로 lapsed 전환"""
    rng = np.random.default_rng(seed)
    for p in policies:
        if p.channel == "GA" and months_since(p.issue_date) >= 25:
            if rng.random() < rate:
                p.status = "lapsed"
                p.lapsed_date = p.issue_date + timedelta(days=25*30)
    return policies

def inject_march_surge(policies, multiplier=1.8):
    """3월 계약 건수를 1.8배로 부풀림 — 균일 분포 가정 대비 주입"""
    # 3월이 아닌 달에서 샘플링해 3월로 재할당하는 방식이 분포 왜곡 없이 깔끔
    ...

def inject_product_outlier(policies, product_id, drop_rate=0.196):
    """특정 상품의 25회차 유지율을 업계 평균 대비 -19.6%p"""
    ...
```

### 이 방식의 면접 방어력
1. *"이상 패턴은 3개 overlay 함수에 캡슐화되어 있습니다. 각 함수는 토글 가능합니다."*
2. *"각 패턴의 파라미터 근거는 KIRI 연구보고서에서 가져왔습니다."* (위 Q5 테이블 인용)
3. *"로드 후 자동 검증 쿼리로 주입이 반영됐는지 확인합니다."* (`SELECT channel, AVG(CASE WHEN months_in_force>=25 AND status='active' THEN 1.0 ELSE 0 END)`)

이 3단 방어가 "데이터 진정성" 질문을 깔끔하게 막습니다.
