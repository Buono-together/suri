3. **이상 패턴 주입은 ‘2단계 방식’으로 하세요.**
   처음부터 생성 로직에 if-else를 섞으면 정합성 디버깅이 힘들고, 패턴 강도 조정도 어렵습니다. 기본 분포를 먼저 만들고, 그 뒤 특정 cohort/product/channel에만 덧씌우는 방식이 이 PoC엔 맞습니다.

# 1) 파일 구조·실행 방식 검토

## 판단

**적절합니다. 다만 한 파일이 빠졌습니다.**

현재 구조:

* `config.py`
* `generators.py`
* `load.py`

이건 24만 행급 PoC에는 충분합니다. 과한 구조는 아닙니다. 다만 `generators.py` 하나에 다 몰리면 금방 더러워집니다.
**`validators.py` 또는 `checks.py`를 추가**하세요.

권장 구조:

* `config.py` — 분포, 시드, 행 수, 패턴 강도
* `generators.py` — 순수 생성
* `mutators.py` — 이상 패턴 덧씌우기
* `checks.py` — 정합성 검증 SQL / assert
* `load.py` — TRUNCATE, COPY, ANALYZE

이유는 단순합니다.
생성 로직과 이상 패턴 로직을 섞으면 나중에 “왜 25회차 유지율이 이 숫자가 나왔지?”를 역추적하기 어려워집니다.

## 실행 방식

`uv run python -m data.seed.load`
이건 좋습니다.

시드 3종 고정도 맞습니다:

* `Faker.seed(...)`
* `random.seed(...)`
* `np.random.seed(...)`

여기에 하나 더:

* **DB 적재 순서까지 deterministic 하게 유지**
  즉, 리스트 shuffle·sample 호출 순서가 바뀌지 않게 해야 합니다.

## 멱등성

TRUNCATE 후 재생성도 맞습니다.
다만 아래 두 개를 추가하세요.

* `TRUNCATE ... RESTART IDENTITY CASCADE`
* 로드 완료 후 `ANALYZE`

PostgreSQL 공식 문서는 대량 적재 후 `ANALYZE`를 권장합니다. 통계가 없으면 플래너가 잘못된 계획을 세워, 시연 쿼리 응답이 흔들릴 수 있습니다. ([PostgreSQL][2])

# 2) 생성 순서와 성능

## 생성 순서

현재 순서:

* products
* agents
* customers
* policies
* premium_payments
* claims

이 순서는 맞습니다.
누락은 없고 FK 의존성도 정상입니다.

## 성능상 최적화

이 규모면 **row-by-row INSERT 하지 마세요. 그냥 COPY 쓰세요.**

PostgreSQL 공식 문서는 대량 적재 시 `COPY`가 대량 INSERT보다 오버헤드가 적고, 대체로 더 빠르다고 명시합니다. psycopg3도 `cursor.copy(...); copy.write_row(...)` 또는 block `write()`를 지원합니다. ([PostgreSQL][2])

### 권장 적재 방식

* Python에서 각 테이블 레코드 생성
* CSV/TSV 임시 버퍼 또는 iterable 준비
* `COPY ... FROM STDIN`
* 마지막에 인덱스 생성 또는 최소한 적재 후 ANALYZE

PostgreSQL 공식 문서는 새로 채우는 테이블이라면 **COPY 후 인덱스 생성**이 더 빠르다고 안내합니다. ([PostgreSQL][2])

### 현실적인 선택

이번 건 24만 행이라 아주 큰 데이터는 아닙니다.
그래서 권장 순위는 이렇습니다.

1. **COPY + 기존 인덱스 유지**
2. 느리면 **TRUNCATE 후 COPY 후 인덱스 재생성**
3. FK까지 떼는 건 이번 규모에선 과함

### 1~2분 목표 기준

이 규모면 로컬 Postgres 15에서 `COPY`로 충분히 1~2분 내 가능성이 큽니다.
오히려 병목은 생성 로직과 Python 객체 생성입니다. 그래서:

* Faker 호출 최소화
* `premium_payments`는 Python date loop를 벡터화/단순화
* payment row를 미리 list로 다 쌓지 말고 **streaming COPY** 추천

# 3) PII 처리에서 추가 고려할 점

## 결론

지금 안은 방향은 맞지만, **주민번호·전화·이메일은 손봐야** 합니다.

## 주민번호

가장 중요한 건 이겁니다.

### 하지 말 것

* “실제 주민번호처럼 보이는 값”을 너무 정교하게 만들지 마세요.
* `rrn_raw`라는 이름도 피하세요.

한국법상 주민등록번호 처리는 원칙적으로 제한됩니다. 당신 데이터가 합성이라 실제 개인정보는 아니더라도, 컬럼명과 포맷이 너무 실제 업무처럼 보이면 괜히 질문을 부릅니다. ([법제처][1])

### 권장

둘 중 하나로 가세요.

* `resident_id_dummy = 90XXXX-AXXXXX`
* `birth_ymd + sex_code_demo + random_token`

즉, **주민번호 “모사”보다 “보험사 내부 식별 토큰처럼 보이는 더미”**가 낫습니다.

### DLP 오탐 방지

체크디짓 오류만으로 부족할 수 있습니다.
DLP는 패턴 매칭만 보는 경우가 많아서, 아예 구분 문자를 바꾸거나 길이를 바꾸는 편이 낫습니다.

예:

* `900101-RRN-483920`
* `19900101-DEM-483920`

이렇게 하면 “생년월일 기반 식별 토큰”처럼 설명 가능합니다.

## 전화번호

`010-XXXX-XXXX`도 실제 번호처럼 보입니다.
면접 데모면 `010-9###-####`처럼 특정 블록을 고정하거나, 아예 `010-DEMO-####` 같은 내부 표현 컬럼을 두는 게 안전합니다.
핵심은 “실제 발신 가능한 번호처럼 보이지 않게” 입니다.

## 이메일

`@suri-demo.local`은 바꾸세요.
`.local`은 mDNS 로컬 네임스페이스와 충돌 여지가 있는 특수 용례이고, 예시용 도메인으로는 `example.com`, `example.net`, `example.org` 같은 예약 도메인이 더 적절합니다. IANA는 이 도메인들을 문서·예시용으로 유지한다고 안내하고 있습니다. ([iana.org][3])

권장:

* `cust000123@example.com`
* 또는 `cust000123@example.org`

## 주소

지금처럼 `시도 + 시군구`까지만 가는 건 좋습니다.
여기서 더 나아가면 오히려 실제 주소처럼 보여서 쓸데없는 질문이 생깁니다.

## 이름

`Faker('ko_KR').name()`은 괜찮습니다.
다만 **동명이인 비율**이 너무 높게 나오지 않게 customer_id를 기본 식별자로 쓰면 됩니다.

# 4) 관계형 정합성에서 누락된 제약

지금 안도 기본 정합성은 맞습니다.
하지만 “자연스럽게 SQL이 나오는 데이터”를 만들려면 아래 룰을 추가하세요.

## 꼭 추가할 것

### 1. policy status와 payment/claim 정합성

현재 `lapsed`만 신경 쓰셨는데, 최소한 아래는 있어야 합니다.

* `active` → 최근 납입 이력 존재
* `lapsed` → lapse 이후 납입 없음
* `matured` 또는 `expired`가 있다면 → 만기 이후 납입 없음
* `surrendered`가 있다면 → 해지환급/중도해지 맥락 설명 가능

지금 상태가 너무 `active/lapsed` 2분법이면 질의가 단조로워집니다.

### 2. 납입주기와 payment_month_seq 일치

`premium_payments_v.payment_month_seq`가 있다면,

* 월납: 1,2,3,...
* 연납: issue_date 기준 12개월 간격인데 seq는 1,2,3...
* 일시납: seq는 1만 존재

이게 안 맞으면 Agent가 “25회차 유지율” 같은 질의를 만들 때 이상해집니다.

### 3. 상품 유형과 claim 발생 가능성

모든 상품에서 claim이 나오는 구조는 부자연스럽습니다.

* 건강/정기/종신 → claims 가능
* 저축/연금 → claim 매우 낮거나 0에 가깝게
* 변액 → 사고보험금보다는 해지·계약변동 쪽이 자연스러움

이 룰이 있어야 “상품별 청구율” 질의가 덜 이상합니다.

### 4. agent tenure

설계사 테이블에 최소한 아래 둘은 있으면 좋습니다.

* `hire_date`
* `branch_code` 또는 `region_code`

그래야 이런 질의가 자연스럽습니다.

* “입사 1년 미만 설계사 판매 실적”
* “지역별 GA 유지율 차이”

### 5. 고객-계약 정합성

고객 평균 3계약은 괜찮지만, 아래 분산이 필요합니다.

* 다수 고객은 1~2계약
* 소수 고객만 4~6계약

평균만 맞추고 전부 3계약이면 가짜 티가 납니다.

### 6. 계약일과 최초 납입일

반드시:

* `first_payment_date >= issue_date`
* 월납이면 통상 issue 후 0~31일 이내 첫 납입
* 연납/일시납은 issue 당일 또는 근접일

## 있으면 좋은 컬럼

* `policy_term_years`
* `payment_term_years`
* `sum_assured`
* `monthly_premium`
* `annualized_premium`
* `lapse_reason_code`
* `claim_type`

이 컬럼들이 있어야 자연어 SQL 질문 폭이 넓어집니다.

# 5) 분포 설계가 한국 생명보험 시장과 얼마나 가까운가

## 결론

**상품유형은 대체로 근사치, 채널은 수정 필요, 성별·연령은 PoC용으로 무난** 입니다.

## 상품유형

당신 안:

* 보장성 55 / 저축성 30 / 변액 15

최근 생보 신규 판매 기준으로는 **보장성 비중이 더 높고, 변액은 더 낮은 편**입니다. 보험연구원 자료에 따르면 2024년 신규 판매 비중은 보장성 60%, 퇴직연금 22%, 저축성 14%, 변액 4%였습니다. 전체 수입보험료 기준은 보장성 49%, 저축성 25%, 퇴직연금 15%, 변액 11%입니다. 

### 수정 권장

당신 스키마에 퇴직연금이 없다면:

* **보장성 60~65**
* **저축성 20~25**
* **변액 10~15 이하**

지금 안의 `변액 15%`는 PoC상 가능은 하지만, “왜 이렇게 높냐”는 질문에 대비가 필요합니다.
방어하려면 “변액은 질의 다양성 확보용으로 의도적으로 과대표집했다”고 말하면 됩니다.

## 상품군

당신 안:

* 건강 35 / 종신 25 / 연금 20 / 저축 10 / 정기 10

이건 **PoC용으로 무난**합니다.
다만 “한국 생보 신규 판매 구조를 더 반영”하려면 종신 비중을 조금 낮추고 건강·보장성 계열을 더 높이는 쪽이 자연스럽습니다. 최근 업계는 IFRS17 이후 보장성 중심 포트폴리오 확대가 분명합니다. 

### 수정 예시

* 건강 40
* 종신 20
* 연금 15
* 저축 10
* 정기 15

## 채널

당신 안:

* 전속 40 / GA 30 / 방카 15 / TM 10 / CM 5

이건 **전속 과대, GA 과소** 쪽입니다.

최근 자료를 보면:

* 2024년 신규 판매는 대리점 39%, 임직원 22%, 설계사 22%, 금융기관보험대리점 16%였고, 
* 별도 자료에서는 생보 개인시장 신계약 건수 기준 GA 비중이 41.3%까지 확인되며, 전속설계사 판매비중은 하락 추세입니다. 

### 수정 권장

`agents`에 임직원 채널이 없다면:

* **GA 35~40**
* **전속 20~25**
* **방카 15**
* **TM 10**
* **CM 3~5**
* 남는 비중은 `direct/employee`를 하나 추가하거나 전속에 일부 흡수

즉, 지금처럼 **전속 40**은 고치세요.

## 성별·연령

* 여 52 / 남 48: 큰 문제 없습니다.
* 30~59 중심: 좋습니다.

보험 가입·유지 분석 시연용으로는 오히려 적절합니다.
다만 20대와 60대도 소량은 넣어야 “연령대별 유지율” 질의가 자연스럽습니다.

권장:

* 20대 8%
* 30대 28%
* 40대 32%
* 50대 24%
* 60대+ 8%

# 6) 이상 패턴 주입 방식 추천

## 후보별 평가

### 1. 단순 룰 기반 if-else

장점:

* 빠름
* 구현 쉬움
* 디버깅 쉬움

단점:

* 패턴이 너무 날것으로 보일 수 있음
* 여러 패턴이 겹치면 정합성 관리가 어렵다

평가:

* 단독으로 쓰면 거칠다

### 2. 가중치 샘플링

장점:

* 분포가 자연스럽다
* 특정 채널/상품의 확률 차이를 부드럽게 줄 수 있다

단점:

* 목표 유지율 68%, 25회차 급락 -19.6%p 같은 **명시 목표치 맞추기 어렵다**
* 면접 직전 수치가 흔들릴 수 있다

평가:

* 현실감은 좋지만 재현성 제어가 약하다

### 3. 2단계 방식 (기본 생성 → 이상 패턴 덧씌우기)

장점:

* 가장 통제 가능
* 기본 분포와 이상 패턴을 분리해 설명 가능
* “왜 이런 숫자가 나왔는지” 역추적 가능
* 면접 리허설과 재생성이 안정적

단점:

* 구현 파일이 하나 더 필요
* 덧씌운 뒤 정합성 재검증 필요

평가:

* **이번 PoC 최적**

### 4. 기타: 점수 기반 시뮬레이션

예:

* lapse propensity score
* claim propensity score

장점:

* 제일 자연스럽다
* 상호작용 효과 반영 가능

단점:

* 7일 남은 상황에서 과함
* 설명이 길어진다

평가:
