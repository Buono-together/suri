결론부터 말하면 **이 PoC는 옵션 A, 공식 MCP Python SDK**가 맞습니다.
당신의 현재 계획인 **fastmcp 선호는 Modify**입니다. 버릴 정도는 아니지만, **금융권 면접 시그널**과 **향후 이관 설명력** 때문에 공식 SDK 쪽이 더 유리합니다.

## 1. 금융권/대기업 면접 컨텍스트

### 판단

**금융권 면접에서는 “공식 SDK를 썼다”가 분명한 플러스**입니다.
이유는 기술 우월성보다 **의사결정 태도**를 보기 때문입니다.

중요한 포인트는 하나입니다.
당신 과제의 핵심은 “빠르게 예쁜 MCP 서버를 만들었다”가 아니라 **거버넌스, 안정성, 책임 추적성**입니다. 금융권은 오픈소스 활용 자체를 금지하지 않지만, 금융보안원은 별도 안내서를 통해 오픈소스 활용·관리 체계와 보안 점검 필요성을 분명히 두고 있습니다. 즉, 문제는 “커뮤니티냐 공식이냐” 자체보다 **왜 그 의존성을 골랐고 어떻게 통제했느냐**입니다. ([금융보안원][1])

### 공식 SDK 시그널

공식 MCP Python SDK는 MCP 프로젝트의 Python 구현이고 MIT 라이선스이며, 현재 stable release 기준 문서가 제공됩니다. MCP 자체도 공식 거버넌스 체계를 갖춘 프로젝트입니다. 이런 조합은 면접에서 “표준에 가깝게 갔다”는 메시지를 줍니다. ([GitHub][2])

### fastmcp를 쓰면 감점인가

**무조건 감점은 아닙니다.**
다만 금융권 면접에서는 보통 이렇게 해석될 수 있습니다.

* 장점: MVP 생산성, 코드 간결성
* 약점: 왜 굳이 래퍼를 하나 더 올렸는가
* 추가 질문: 장애 시 어디까지 직접 통제 가능한가

즉, fastmcp를 써도 “실용적 선택”으로 방어는 가능하지만, **면접관이 추가로 캐묻기 쉬운 선택**입니다.

## 2. 프로덕션 이관 가능성

### 판단

**이관 관점에서도 공식 SDK가 유리합니다.**

fastmcp는 Apache 2.0이고, 공식 SDK 의존성 `mcp>=1.8.0,<2.0.0` 위에 올라가는 래퍼입니다. 즉 fastmcp를 고르면 의존성이 줄어드는 게 아니라 **한 층 늘어납니다.** ([GitHub][3])

금융권 보안 심사에서는 Apache 2.0 자체가 문제라기보다,

* 외부 의존성 수
* 유지 주체
* 취약점 대응 체계
* 버전 고정과 SBOM 관리
  이런 항목이 더 중요합니다. 금융보안원 안내서도 오픈소스 활용을 전제로 하되 관리 체계와 보안사고 예방을 강조합니다. ([금융보안원][1])

### 실무적 의미

공식 SDK 직접 사용은 이렇게 설명하기 쉽습니다.

> “MCP 표준 구현체를 직접 사용했고, 도메인 제약은 애플리케이션 레이어에서 명시적으로 통제했습니다.”

반대로 fastmcp는 이렇게 됩니다.

> “공식 SDK 기반 래퍼를 사용해 개발 속도를 높였습니다.”

이 답변도 가능하지만, **사내 이관 문서**를 쓰는 순간 공식 SDK 쪽이 더 단순합니다.

## 3. 장기 유지보수 리스크

### 판단

**장기 유지보수는 공식 SDK가 더 안전합니다.**

공식 MCP Python SDK는 MCP 프로젝트의 공식 Python SDK이고, MCP 자체는 공식 거버넌스 문서를 가진 프로젝트입니다. 반면 fastmcp는 빠르게 발전하는 고수준 프레임워크이지만, 별도 추상화 계층이며 공식 스펙 구현체 위에 올라갑니다. ([GitHub][2])

여기서 중요한 건 “1인 프로젝트냐 아니냐”보다 **추상화 계층 추가 비용**입니다.
래퍼는 편하지만, 아래 리스크가 있습니다.

* 공식 SDK 변경 → fastmcp 대응 lag 가능
* fastmcp 고유 API 변경 → 코드 수정 필요
* 장애 시 원인층이 하나 더 늘어남

반대로 공식 SDK는 boilerplate가 더 많지만 **바로 spec/SDK 레벨에서 이해하고 대응**할 수 있습니다.

### SURI 이후 재사용

6~12개월 포트폴리오 재사용까지 보면,
**공식 SDK 경험이 더 보편적 자산**입니다.

## 4. MCP 스펙 변경 대응력

### 판단

**공식 SDK 우위**입니다.

MCP 공식 스펙은 계속 진화 중이고, 공식 사이트에 최신 스펙과 로드맵, 거버넌스 체계가 공개돼 있습니다. 스펙의 source of truth는 TypeScript schema라고 명시되어 있고, 공식 Python SDK는 그 표준 구현 계열입니다. ([모델 컨텍스트 프로토콜][4])

fastmcp도 현재는 활발하고 생산성이 높지만, 구조적으로는 공식 SDK 위에 한 겹 더 있는 형태라 스펙 변경 시 **즉시 추종성**에서는 공식 SDK보다 불리할 수 있습니다. fastmcp 쪽도 “schema generation”, “transport negotiation”, “protocol lifecycle”을 추상화한다고 밝히고 있는데, 이건 편한 대신 내부 추상화가 스펙 변화의 완충층이 된다는 뜻이기도 합니다. ([GitHub][5])

### 9일 내 완성 기준

7일 남은 지금은 둘 다 완성 가능합니다.
하지만 질문은 “6~12개월 재사용”까지 포함했으니, 여기서는 **공식 SDK 승**입니다.

## 5. 디버깅·운영 투명성

### 판단

**공식 SDK가 더 투명합니다.**

금융권에서 중요한 건 “누가 잘못했는가”가 아니라
**어느 레이어에서 실패했는지 빨리 특정할 수 있느냐**입니다.

SURI는 이미 3층 방어 구조가 핵심이고, MCP 서버는 Layer 1입니다.
이때 공식 SDK를 쓰면 장애 분석 흐름이 단순합니다.

* MCP 프로토콜/transport 문제
* tool schema 문제
* SQL validation 문제
* DB role/schema 문제

반대로 fastmcp를 쓰면 여기에 **프레임워크 추상화 레이어**가 하나 더 들어갑니다.

이건 치명적 약점은 아니지만, 면접에서는 다음 질문을 부릅니다.

* 자동 생성된 schema가 기대와 다르면 어떻게 검증했나
* decorator 추상화 안쪽 동작을 어디까지 이해하나
* spec mismatch 시 어디를 먼저 보나

도구가 1~3개뿐인 MVP라면, 추상화로 얻는 이익보다 **명시성**이 더 값집니다.

* 7일 남음

### 판단

**ROI 기준으로도 공식 SDK가 더 낫습니다.**

보통 fastmcp는 도구가 많아지고, schema 자동 생성과 편의 기능에서 효율이 커집니다.
그런데 당신은 tool 1~3개입니다. 이 정도면 공식 SDK의 boilerplate가 치명적이지 않습니다.

즉,

* fastmcp의 이점: 개발 속도
* 공식 SDK의 이점: 면접 메시지, 이관 설명력, 운영 투명성

이번 과제에선 후자가 더 큽니다.

# 면접 방어 답변 템플릿

## “왜 공식 SDK를 선택했나요?”

> SURI는 MCP 기능 자체를 과시하는 프로젝트가 아니라, 금융권 환경에서 자연어 SQL 접근을 어떻게 통제할지 보여주는 PoC였습니다. 그래서 생산성보다 표준 구현체 기반의 명시성과 운영 투명성을 우선했고, tool 수가 1~3개라 공식 SDK의 boilerplate도 충분히 감당 가능하다고 판단했습니다. 또 추후 사내 이관이나 보안 검토를 생각하면 의존성 레이어를 줄이는 편이 설명이 쉬웠습니다.

## “왜 fastmcp를 쓰지 않았나요?”

> fastmcp는 좋은 선택지라고 봤습니다. 다만 이번 PoC는 도구 수가 적어서 자동 schema 생성의 이점이 크지 않았고, 대신 금융권 면접에서는 공식 구현체 사용이 더 보수적이고 설명 가능한 선택이라고 판단했습니다. 추상화 한 층을 줄여 디버깅과 책임 경계를 명확히 하는 쪽이 이번 과제 목적에 더 맞았습니다.

## 반대로, fastmcp를 이미 일부 써버렸다면

> MVP 속도를 위해 초기에 fastmcp를 검토했지만, 최종 제출본에서는 공식 SDK 기준으로 정리하는 것이 금융권 PoC 목적에 더 맞다고 판단했습니다. 이번 과제는 MCP 서버 개발 편의성보다 거버넌스와 통제 가능성이 핵심이었기 때문입니다.

# 금융권 특화 규제 관점에서 한 줄

개인정보보호법, DLP, 망분리 이슈가 있다고 해서 fastmcp가 직접 금지되는 건 아닙니다.
하지만 금융권은 오픈소스 사용 시 **보안성 점검, 관리 체계, 책임 추적성**을 중시하므로, 그런 맥락에서는 공식 SDK 직접 사용이 더 방어하기 쉽습니다. 주민등록번호 등 고유식별정보 처리는 여전히 엄격히 제한되므로, MCP 레이어에서도 read-only·schema 제한·AST 검사처럼 **통제 가능성이 명확한 구조**를 보여주는 게 중요합니다. ([금융보안원][1])
